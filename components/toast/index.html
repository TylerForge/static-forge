Toast

Toasts provide brief messages about app processes at the bottom of the screen.

Interactive demo

Guidance

Toasts inform users of a process that an app has performed or will perform. They appear temporarily, towards the bottom of the screen. They shouldn’t interrupt the user experience, and they don’t require user input to disappear. 

Best practices

  • Only one toast may be displayed at a time. Use successive toasts if more than one is needed. 
  • A toast can contain a single action. Because they disappear automatically, the action shouldn’t be “Dismiss” or “Cancel.”
  • If there is a FAB on the page, toasts should appear above the FAB.

Do.

If an action is long, it can be displayed on a third line.

Don’t.

If an action is long, it can be displayed on a third line.

When to use

Toasts communicate messages that are minimally interruptive and don’t require user action.

Component Priority User action
Toast Low priority Optional: Toasts disappear automatically
Banner Prominent, medium priority Optional: Banners remain until dismissed by the user, or if the state that caused the banner is resolved
Dialog Highest priority Required: Dialogs block app usage until the user takes a dialog action or exits the dialog (if available)

Client-Facing Terminology

Do not use the terms snackbar or toast in a client-facing context.
Refer to these as messages or notifications.

Example: The program displays a message showing the item number.

Omnibar

The omnibar is a user’s personal navigation kit. It holds everything a user needs to navigate the app they’re currently in, apps they typically go to, and apps they might need, as well tools to customize their navigation experience through notifications, language options, and help resources.

Client-facing terminology

Note: Do not capitalize omnibar in written documentation.

Guidance

The omnibar spans the top of an app and is a user’s personal navigation kit. It holds everything a user needs to navigate the app they’re currently in, apps they typically go to, and apps they might need, as well tools to customize their navigation experience through notifications, language options, and help resources.

Anatomy

The omnibar is comprised of five key sections:

  1. Internal navigation (Optional). See Navigation Drawer for more detail.
  2. Branding and app title (Required)
  3. Search (Optional)
  4. Action items (Optional)
  5. Persistent user info (Required)
Desktop

On desktop, all five sections are visible in the omnibar.

Omnibar sections

Let’s take a look at those five key sections: Internal navigation, branding and app title, search, action items and user info.

1. Internal navigation

The menu or hamburger icon controls an app’s primary level of navigation. Apps using a navigation drawer should use a permanent drawer by default, unless a specific business need requires otherwise (ie, kiosk app or similar). On mobile, use a dismissible drawer that is closed by default. See Navigation Drawer for more information.

2. Branding and app title
Backoffice apps

Backoffice apps use the standard Indigo 800 as a background for the omnibar. The backoffice omnibar features the Tyler Talking T’s logo and the app name, in title case (“General Ledger,” not “General ledger”).

Public facing apps

For citizen-facing applications, clients may choose to theme their omnibar using a preset selection of Material colors. Text, search inputs, and icon color are adjusted to meet appropriate ADA color contrast levels.

The public-facing omnibar features the client logo (32px by 32px) and municipality / app name appear just to the right; separated by a divider. Both are title case (“Buena Vista | My Utilities”).  

On mobile, the public-facing omnibar uses the prominent attribute to accommodate the municipality name and app title.

3. Search
Search scope

No search

Use when an app doesn’t have a use case to search for data and global search doesn’t need to be supported.

Global search

Use when there’s no use case to search for data within the app, but users may need to search for something within the Tyler ecosystem. Include the “global icon” at the end of the search.

Internal app search 

Use when searches within a page, app or product suite are needed. Common examples include: searching for a user, record, incident, case, or setting. Use placeholder text to indicate that the search is contextual within the app.

Combined search 

Use this to allow searches at multiple levels – within a page, app, product suite, OR within the Tyler Ecosystem. Use placeholder text to indicate where the search will occur.

Search input type

One search input 

Default search state for the omnibar. Use in most cases.

Multi-search input

Use where multiple fields are required to run a single search.

Advanced search

Use when a more advanced search is required for more than 30% of searches.

Pre-filtering and autocomplete

Autocomplete

Use to suggest exact results or suggested search queries.

Prefilter

Use to help users search on specific categories or to improve performance time for global searches.

4. Action items

Up to three global actions maybe be placed in the “Action items” section. Actions belong to the app omnibar if they affect the application or record as a whole. Actions that affect a single item or a piece of the application should be placed in context, close to the part they affect. 

Select from the following common actions such as “Shopping cart” and “Export all.” Additionally teams can create custom actions to meet specific business needs such as “Print package,” etc.

If your applications has more than three action items, place the two primary actions in the omnibar and the additional actions in an overflow menu.

Language switcher

For most apps, the language switcher lives in a user’s profile. For kiosk apps and apps geared toward an unauthenticated experience, the language switcher may appear as an action item. 

Do.

For apps with more than three action items, place the two primary actions in the omnibar and the additional items in an overflow menu.

5. User info

The final section of the omnibar contains information and actions pertinent to a specific user. User information consists of four areas. 

  1. Help
  2. Notifications
  3. App launcher 
  4. Avatar / Profile card
Help

The help menu is standard across apps and consists of a variety of resources, ranging from help specific to the user’s current app to the broad range of resources available on Tyler Community. 

Notifications

Notifications are standard across apps, and use tabs to display notifications from the current app as well as a summary of notifications from all apps. 

Example notifications include:

  • “New time off request to approve”
  • “New package rejected”
  • “Time off request approved.”
  • “Your bill is due in 5 days.”

A badge displays the total number of unread notifications individual to the user. Notifications are considered “read” when they have been clicked.  

Notifications may be clickable. Clickable notifications may deep link within an app or may link to an external resource such as a Tyler Community article. Read notifications should persist only for 14 days unless otherwise specified by a business requirement. 

App launcher

The app launcher serves as a user’s local map to their Tyler ecosystem. For more information, read about the app launcher here.

Avatar and profile card

The profile card features a user’s personalized avatar if they’ve set one, their full name, and the email used to sign up with the app. Additionally, the card features a “Sign out” option as well as a link to their profile.

Backoffice apps

Backoffice apps support a user profile dialog, which features the following editable information:

  • First and last name
  • Language preference
  • Avatar photo

Language preference

By default, the language selector should detect the native browser language if possible. Otherwise, users can select from supported languages in from their profile. 

Public facing apps

Public facing apps support a user profile. In public facing apps, users can access their global preferences from their profile page, including:

  • Language preferences
  • Contact info
  • Payment methods

For more information, see the Profile guidance (Coming soon).

Unauthenticated users

For unauthenticated users, the avatar is replaced by a “sign in” button. Notifications are hidden; the help menu and app launcher remain.

Contextual states

The omnibar can transform into a contextual action bar to provide contextual actions to selected items. For example, when records are selected in a list, the omnibar transforms to a contextual app bar with actions related to the selected records.

When the omnibar transforms into a contextual action bar, the following changes occur:

  • The bar color changes
  • The navigation icon is replaced with a close icon
  • The omnibar title text converts to contextual action bar text
  • All omnibar actions are replaced with contextual actions

Upon closing, the contextual action bar transforms back into the omnibar.

Select

Desktop 

Apps that primarily consist of a list or table of records may use the omnibar for selection. Don’t use the omnibar for selection when a page has more content than just a list or when the omnibar is far from the selected content.

The selection state includes a checkbox to select or unselect all records, and a sum of total records selected. By default, “select all” includes all records currently visible on the page. When pagination or infinite scroll is used, provide users with an option to select all available records. 

Actions that can be taken on selected records are placed at the end of the omnibar’s selection state. Use icons and a tooltip for familiar actions (Edit, Delete, Mark complete, Print, Download). Use outline buttons for less familiar actions (Assign to, Add role, Add to group). 

Don’t.

Don’t combine text buttons and icons.  

Don’t.

Don’t use the omnibar for selection when a page has more content than just a list or when the omnibar is far away from the selected content.

Mobile 

On mobile, the contextual select state consists of:

  • An option to close the selection state 
  • A count of select records 
  • Contextual actions represented by icon buttons
Focused search

Desktop

For search-oriented apps, teams may implement a focused search state. Focused search can draw additional attention to a search mode and can be useful for search-driven applications, but is not required. Focused search state may be combined with any of the search variations highlighted above. 

Mobile

On mobile, the Search icon expands into a full search input on tap. Users can tap the Close icon to return to the default omni state.

Omnibar variants

Tabs
Top tabs can be placed within the omnibar as primary navigation when the navigation drawer is not being used. Use 128px height app bar, with tabs placed along the bottom edge.

Best practices

Don’t.

Don’t place links in the omnibar. Use the tabs navigation, if necessary.

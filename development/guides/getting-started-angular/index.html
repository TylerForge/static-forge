Getting Started - Angular

This guide will walk you through how to consume the TCW library within an Angular application. The following guide assumes that you have created an Angular application using the Angular CLI (version 8.x at the time of writing, but should pertain to older versions as well).

If you’d like to reference a real demo Angular application while following along with this guide, feel free to browse and pull down the code in the TCW Angular Demo application on bitbucket. 

If you’re using the dapper template, there is also a reference application for that located here.

Configuration

If you haven’t already done so, create an .npmrc file in the root of your Angular project with the following content

registry=http://yarpvproget/npm/npm_Tyler_Core/

Note: If you’re using the npm_TEG_Foundation_Web feed currently, there is a connector used behind the scenes on npm_Tyler_Core to allow access to packages in the other feed. You can safely use npm_Tyler_Core only to ensure that you still get access to existing packages on npm_TEG_Foundation_Web while still getting access to all new Tyler-wide core npm packages.

If you are already using a custom registry in your .npmrc file for a different feed, you can add another line that will scope just the @tylertech and @tyler-components-web packages:

@tylertech:registry=http://yarpvproget/npm/npm_Tyler_Core/
@tyler-components-web:registry=http://yarpvproget/npm/npm_Tyler_Core/

Installation

Install the TCW component library package as a dependency for your project.

$ npm install @tylertech/tyler-components-web

The next step is optional, but for a more seamless experience you can (and should) install the TCW Angular adapter library package. This package provides Angular services and directives to ensure that working with the Web Components is a familiar experience for an Angular developer.

Important: This package makes things like ngModel, formControlName, and formControl work on elements that Angular doesn’t know how to bind to.

Install the package using the command below.

$ npm install @tylertech/tyler-components-web-angular

Then you need to import TylerComponentsWebModule in any Angular module that you need it in.

import { TylerComponentsWebModule } from '@tylertech/tyler-components-web-angular';

@NgModule({
  imports: [
    TylerComponentsWebModule
  ]
})
export class AppModule {}

Note about versioning: The @tylertech/tyler-components-web-angular major version always maps to the major version of Angular that is is compatible with. For example, if you’re using Angular version 5.x, you will want to install the latest 5.x version of @tylertech/tyler-components-web-angular… etc. and so on for 8.x and 9.x.

Important: By default, npm will always install the latest version of a package. If you’re using an older version of Angular, be sure to check that the major version matches, otherwise you may run into incompatibility errors.

Enable the CUSTOM_ELEMENTS_SCHEMA

To enable the usage of custom element tags to be used within your Angular templates, you need to tell Angular that you plan to use them so that it can ignore element tags that it doesn’t know about.

import { NgModule, CUSTOM_ELEMENTS_SCHEMA } from '@angular/core';

@NgModule({
  schemas: [
    CUSTOM_ELEMENTS_SCHEMA
  ]
})
export class AppModule { ... }

When using CUSTOM_ELEMENTS_SCHEMA you are essentially telling Angular to ignore enforcing some of its built-in element/property/attribute validation. This allows Angular to skip validating these elements, and therefore will not report certain useful errors at runtime if you bind to the wrong properties, or forget to import the components. Keep this in mind as you use Web Components with the Angular and be more mindful about your syntax when using them.

Usage

When consuming the library, it is best practice to import the TCW components that you are using individually. This is done mainly to support tree shaking to ensure that we aren’t bundling extra component code that we aren’t using from the library.

Open up the main.ts file (or whichever file contains application bootstrapping), and import each component from the library. Then you need call defineCustomElements and pass the array of components that you want to define.

import { defineCustomElements } from '@tyler-components-web/core';
import { AppBarComponent, ScaffoldComponent } from '@tylertech/tyler-components-web';

defineCustomElements([
  AppBarComponent,
  ScaffoldComponent
]);

platformBrowserDynamic().bootstrapModule(AppModule).catch(err => console.error(err));

Here we are importing the defineCustomElements function from the @tyler-components-web/core library, and we use that to tell the browser to register the AppBarComponent and ScaffoldComponent components as custom elements. This internally calls the window.customElements.define() API.

This could could technically go anywhere in your application. Since the TCW components do not rely on Angular, we just need to register them somewhere, we can then use the HTML tags for these custom elements anywhere in our Angular app just as we would any other HTML element like a

or for example.

By importing each component individually, you are then allowing the tree shaking process during a production build to avoid including unused code from the component library.

Another option you have is to import every component from the library if you know that you will be using everything. This is done by importing and calling a convenience method from the library called defineComponents.

import { defineComponents } from '@tylertech/tyler-components-web';
defineComponents();

The library provides this function as a convenience for developers to easily import and register every component in the library without having to specify each component individually. Keep in mind that this will cause every component from the library to include all of its code in your bundle. This may have unwanted side-effects related to performance if you aren’t careful.

Choose the approach that works best for your application, while keeping in mind what code is being downloaded at runtime.

Global Stylesheet

This step will guide you through the process of adding the TCW global stylesheet (tcw-material-design.css) to your application.

Most of the components in the TCW library are self-contained, meaning they encapsulate their own JavaScript, CSS, and HTML. There are certain components (The button and table components for example) that do not follow this pattern, and therefore rely on the styles being available globally on the page.

To ensure that you get the proper styling in place, please follow one of the methods below. Note: These steps assume you’re using the Angular CLI to build your application.

Option #1

Add a reference to the stylesheet in your index.html file as a tag.

Option #2

Add the file to your angular.json configuration file so that it includes global styles in your app during the build.

Option #3

Import the stylesheet in your global styles.scss file.

Option #4

Import the stylesheet in your app.component.scss file and enable ViewEncapsulation.None in your app.component.ts file.

Fonts and Icons

Add a link tag that references the Roboto font (with desired font weights), and MaterialIcons font to allow for properly setting up our typography and iconography:


rel="stylesheet" href="//fonts.googleapis.com/css?family=Roboto:400,500,700,400italic|Material+Icons">

Install Polyfills

The TCW library relies on the official web component polyfills to support older browsers. Currently IE 11 and Edge do not support the Web Component spec natively. To ensure that the components work in those browsers run the following command to install the Web Component polyfills.

$ npm install --save @webcomponents/webcomponentsjs

Important note: Even if you’re not planning on supporting IE 11 and Edge, the installation of this package is still required for the following step.

Include custom-elements-es5-adapter.js

Due to the fact that the TCW library ships its code in ES5, a separate (very lightweight) polyfill is required to make the ES5 components compatible with browsers that support custom elements natively (Chrome, Firefox, Safari… etc.).

Until we are able to stop supporting IE 11, we will continue to downlevel our code to ES5 for the most wide coverage of browsers. When that requirement is removed, we will then update the packages to support ES6 and up, at which point these polyfills will not be needed at all.

Open your polyfills.ts file, and add the following line:

import '@webcomponents/webcomponentsjs/custom-elements-es5-adapter.js';

Per the custom elements spec, all custom elements must be a class that extends HTMLElement. Once all browsers natively support custom elements, we can safely ship the libraries in ES6 and remove the need for this polyfill.

(Optional) Include Web Components Polyfill

If you need to support a browser that does not natively support custom elements (IE 11 and Edge), then the final step of requiring the standard, official, web component polyfill is required.

Once again open the polyfills.ts file, and add the following line:

 import '@webcomponents/webcomponentsjs/webcomponents-bundle.js';

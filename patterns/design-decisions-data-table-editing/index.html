Design decisions: Data table editing

Data tables will use a custom editing component in order to facilitate keyboard first, heavy data entry.

Status:
Design: In progress      Web component: In progress

Initial business requirements

Users can:

  • Edit a single cell 
  • Add, edit, delete a single table row 
  • Edit a full data table 
Considerations for a better table experience. 
  • Provide error forgiveness (cancel / undo at the cell or row level )
  • Facilitate bulk editing
  • Support keyboard first (tab) interactions
  • Provide visual affordances that row or values within row are editable
  • Allow for long entry / overflow 
  • Provide for single cell / full row error state
  • Display text overflow on hover or click, as needed
 

Reasons to diverge from Material guidance

Material’s guidance 

Material design recommends a small or large edit dialog for users to modify or add table information.

While this pattern provides focus to infrequent edits within a table, it becomes distracting and cumbersome from heavy data entry; obscuring other cell values and providing unnecessary animation. 

Instead we’re proposing inline editing that uses the dense TCW bordered inputs against an active state within the table row. The pattern is intended to create a seamless editing experience to facilitate a state of flow when updating potentially large amounts of information. An editing state will be triggered explicitly by users at row or table level. Edit mode will facilitate both mouse and tab interactions.

Initial design plan

5/14/2019 (Nick, Kieran, Emily)
Supported editing states:
  • Single cell (uses pencil icon to enable editing in a cell; within editing state, displays save / cancel within cell)
  • Row (uses pencil icon to enable editing in a row; within editing state, displays save / cancel at end of row)
  • Full table (trigger edit from table header; within edit state, bulk action header can be used to hold save / cancel; teams can decide whether to support automatic or explicit saving)
  • Dialog or sidesheet editing (use for more information than displayed in the cell)
 
Editing affordances
  • Explicit edit (pencil) EITHER on hover or persistent
    • at cell, row, table level
Style
  • On edit, use filter inputs and apply row background color (active state; indigo @ 10%)
  • Row inputs support text, dropdowns, datepicker, switches, icons
  • Display save / cancel (at cell level or row level)
    • at whatever level you’re working at? (per cell, per row, auto save, save at table level
Validation
  • Use default input validation styling
    • Options for error text: inline text beneath field, tooltip for fewer characters, overlay/modified dialog for longer messages)
  • Use same styling for hints
Overflow
  • Later on: add resizable columns
  • Tooltips? modified dialog? help on hover?
    • click into field for ellipsis with an overlay?
    • Revisit this?
Questions to answer in the future
  • Can the full table be toggled to be editable or should this be handled row by row?
  • revisit overflow styling?
  • Consider Handsontable component for heavy editing needs

Feedback session - 6/10

  1. Met with representatives from Tyler product teams to review designs so far. 
Concerns with current designs:
  • Where should records be added?
    • Some concerns about findability for adding records to the bottom of a table 
    • Consider allowing users to records to a specific location within a data table
    • How should “rows per page” be handled as a user adds a new record? Should it update to reflect a nonstandard number of rows?
    • Follow up on this in UX research
  • Full data table edit should provide options for either automatic or explicit save
    • We can recommend automatic saving for full table editing to improve data preservation
    • But also provide option for teams to use an explicit “save” and “cancel” to increase user control, especially in administrative contexts
 
Other use cases to consider:
  • Datatables within GL / PO’s / financial products can get more complex than what’s covered here 
  • Allow for conditional logic within adding 
    • ie, selecting a specific value from a dropdown enables or disables specific columns within the row entry 
    • Dialogs could be too cumbersome make these entries quickly / efficiently. 
  • Consider keyboard shortcuts / interactions used by legacy products so we can be as consistent as possible
    • Let’s set a defined set of keyboard interactions and follow up in UX testing. 
    • Ie, should “enter” save data or create a new row?
  • Consider supporting column or row sums 
    • How will this be reflected in inline editing pattern?
    • Potentially recommend HandsOnTable for an Excel-like experience when advanced calculations are needed beyond a simple column sum
    • Loot at Swimlane table example for built in column sum 
  • Consider data density. 
    • If we support “compact” and “roomy” styles, consider impacts on inline editing.
    • What is our row height minimum at which to support editing?
    • May depend somewhat on final text field styles / heights.
  • Consider these three guiding “tiers” instead of frequency tiers potentially?
    1. Non-prescriptive data entry (works well with current inline editing model)
    2. Prescriptive / specific data entry (ie, GL and PO examples which may include conditional entry/validation and more complex interactions)
    3. Bulk editing – supporting quick data entry for use cases where large amounts of data are added/edited at once

Updated designs: Kathy's review

5/28/2019, updated guidance in bold

**Consider tiers for table guidance:

1. Where editing is infrequent (less than 25% of the time), viewing is the primary action (80%+ of use), and records are edited singly, use table with single row edit pattern. 

2. Where users are equally likely to view and to edit, and records are likely to be edited in bulk., use full table editing pattern.

3. Where editing is the primary action (75%+ of the time), use the material skin of HandsOnTable to closely mimic the Excel data entry experience. 

Inline row edit

Table cells become standard dense TCW bordered inputs.

Row background becomes primary active state.

Currently focused input receives standard TCW focus state.

Row actions become “save” and “cancel”; both apply to the row in full.

“Cancel” undoes any changes made and reverts data.

Checkboxes and row actions become disabled or hidden so that users can’t take additional actions in edit mode.

Actions should have a hover tooltip: “Save or cancel edits first.”

Single cell edit

Uses the same styling as row editing for consistency, but only fields that can be edited become standard dense TCW bordered inputs. 

Row actions become “save” and “cancel”; both apply to the row in full.

Full table edit

User hits “edit” to enable edit mode.

In edit mode, header changes to display edit actions, using same style as bulk action header. 

First editable field receives focus; user can click or tab any field as needed. 

Recommend that full table use automatic save to prevent data loss. 

Automatic save mode should indicate that changes have been saved and offer a way for users to undo. 

User hits “save” or “finish” to close edit mode. 

Disable actions in full table edit mode, with tooltip guidance (see above).

Adding a row

For tables that allow adding a new row, “Add” should be placed in the footer, close to where new row will be added. 

New row uses single row edit styling; all inputs should be empty. 

“Cancel” deletes the new row. If data has been entered, user is prompted with a dialog to discard or keep changes. 

Users hits “save” or enter to add row and will see a toast confirmation. Row highlight persists for a few seconds and fades to show where the row has been added. 

Recommend that new rows be added to the bottom of a table; if app team is using infinite scroll, recommend that rows be added to the top of the table instead. “Add row” displays at the top of the table accordingly. 

For tables where adding new records is the most common action, use an extended fab. By definition, a represents the primary action of a screen, so it should only be used in contexts where adding is the primary action. Otherwise, use a flat button for the “add new” action.

Error validation

Inputs with errors should display an error icon within the field. On hover, user sees a tooltip with the error. 

Input takes the default TCW error border. 

Validation should occur as the user tabs to the next field.

Tooltips should be used for validation under three words.

For validation messages longer than three words, use a modified toast on hover to improve readability. 

Toast should be text only.

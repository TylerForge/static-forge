Forms

Forms are used in almost all Tyler products. This page should give you a sense of how individual components (toggles, checkboxes, data fields) are used collectively to create a consistent “form” pattern, as well as how to make forms ADA compliant and intuitive.

Guidance

Forms are used for entering and editing data. 

Forms may utilize the following components. Visit the individual component pages for in-depth details of specific states and visuals.

  • Typography standards
  • Data formatting
  • Text fields
  • Selection controls
  • Buttons (for submission)

Form layouts

Desktop

Vertical

Use this layout when inputs are grouped in a way that encourages users to read or tab down vertically, one column at a time. 

When a screen is sized down, columns should stack on top of each other. 

Use dense inputs, 16px of vertical spacing between inputs (32px between inputs without helper text), and 48px of horizontal spacing between columns. 

Horizontal

Use this layout when inputs are grouped in a way that encourages users to read or tab horizontally. 

When a screen is sized down, fields should wrap accordingly. 

Use dense inputs, 16px of vertical spacing between inputs (32px between inputs without helper text), and 24px of horizontal spacing between columns.)

Mobile

Vertical

Use this layout when inputs are grouped in a way that encourages users to read down vertically, one column at a time. Use this for tablet or portrait mobile sizes.

When tablet screens are rotated to portrait mode, fields should wrap accordingly. 

Use roomy inputs, and 24px of vertical space between inputs (40px between inputs without helper text) and 32px of horizontal spacing between columns. 

Note: For traditional phone sizes in portrait mode (screen width between 350px and 480px) fields should be full width and stack vertically.

Horizontal

Use this layout when inputs are grouped in a way that encourages users to read across horizontally. Use this for tablet or portrait mobile sizes. 

When tablet screens are rotated to portrait mode, fields should wrap accordingly. 

Use roomy inputs, and 24px of vertical space between inputs (40px between inputs without helper text) and 16px of horizontal spacing between columns. 

Note: For traditional phone sizes in portrait mode (screen width between 350px and 480px) fields should be full width and stack vertically.

 

Best practices

All forms are comprised of 7 elements.

1. Labels

Do.

Use succinct, natural language and consistent capitalization.

Do.

If most fields are optional, indicate which are required using an asterisk.

Do.

If most fields are required, indicate which are optional with “(Optional).”

2. Input fields

 Input fields can take a variety of forms including text fields, dropdowns, radio buttons,  checkboxes, and dropdowns.

Do.

The length of input fields should reflect their expected content.

Otherwise, use a consistent length that provides enough room for correct answers.

Do.

Use auto-formatting for fields such as phone number and credit card number to reduce validation errors and format uncertainty. Constrain data inputs such as phone numbers and credit card numbers to numbers only.

Don’t.

Don’t cut off text by using a field that’s too small for the intended input.

Do.

Ensure that inputs grow with text as appropriate. Use a textarea  control to accommodate longer text entry.

3. Placeholder text

Where possible, use assistive text below a field instead of placeholder text. According to Norman Nielsen, placeholder text can pose issues because of its transitory nature and can make it difficult for users to remember what information belongs in a field. 

Caution.

If you need to provide additional information to users, consider using assistive text instead of placeholder text where possible. In the above example, “ie, todd@example.com” could displayed as helper text instead.

When using placeholder text, ensure that the label floats persistently. 

Don’t.

Don’t use placeholder text as a label. Since placeholder text disappears on user input, use a label element to ensure that the field label persists.

Do. When using placeholder text, ensure that the label floats persistently.

4. Actions

Actions allow users to submit a form. “Save” and “Cancel” are the most standard. 

In general, forms should have one or two actions.

Primary actions – the action a user most likely to take – should have higher visual emphasis, while secondary actions should have less. 

Do.

Do use a higher visual emphasis for primary actions.

Caution.

In general, provide users with the two most likely actions. Avoid potentially overlapping actions (“start over” and “cancel”).

Don’t.

Don’t use two raised buttons next to each other in a form. It inappropriately communicates the same visual emphasis for actions that are unequally likely.

Don’t.

Don’t use two raised buttons next to each other, even if only one has the primary background color.

5. Validation

 Error text ensures the data submitted by the user conforms to acceptable parameters.

Do.

Provide actionable remedies that will help users enter correct information.

If only one error is possible, error text describes how to avoid the error.

If multiple errors are possible, error text describes how to avoid the most likely error.

Don’t.

Validate information before a user has entered information. Validate when a user leaves a field.

6. Defaults

Defaults can save users time and effort. 

Use defaults whenever reasonable predictions about users’ goals can be made. 

When using radio buttons, provide a default selected option if possible.

Do.

Default in today’s date when applicable. 

7. Helper text

Provides assistance on how to fill out a field. Helper text is optional.

Use when:

  • Forms ask for unfamiliar data.
  • People question why they are being asked for specific data.
  • There is concern about data privacy/security.
  • Certain data fields are optional or required when the rest of the form is not.

Do.

Use concise and specific guidance to recommend ways of answering and prevent errors. 

Do.

Inform users how their personal data will be handled.

Errors

Errors occur when an app fails to complete an expected action. Errors can occur when a user input is not understood, an app fails to load, or incompatible operations are run concurrently.

Introduction

Errors can be categorized into three groups:

  • User input errors (ie, user enters invalid data into a specific field).
  • General user errors/warnings (errors initiated by the user that may require follow up actions on their part to correct).
  • Page errors (server exceptions such as 404s and 500s).

User input errors

Error message
When text input isn’t accepted, an error message can display instructions on how to fix it. Error messages are displayed below the input line, replacing helper text until fixed.

Best practices
  • Provide as much information up front as possible. Use helper text to alert users to possible errors.
  • If only one error is possible, use error text to describe how to avoid the error; if multiple errors are possible, use error text to describe how to avoid the most likely error.
  • Validation should occur when focus leaves a field.
  • Persist values; only clear fields when dealing with credentials or sensitive information.
  • In general, “Submit” or “Next” should be disabled until an error is addressed. Don’t allow the user to proceed if required fields contain errors.

Caution.
Long errors can wrap to multiple lines if there isn’t enough space to clearly describe the error. In this case, ensure padding between text fields is sufficient to prevent multi-lined errors from bumping layout content.

General errors

General errors are initiated by the user and may require follow up action to correct.

Errors on submit

In general, validation should occur on specific fields as soon as the user navigates to the next field. In cases where validation must occur after a form is submitted, display a message at the top of the form or screen summarizing the fix(es) needed and any additional explanation. If specific inputs need to be corrected:

  • Link to the invalid field in the error summary
  • Indicate the fields where a fix is needed by displaying error text below the field.

Displaying errors

Depending on their urgency, errors may be displayed as toasts, banners, dialogs, or page errors. 

 
Component Urgency Content
Banner Prominent, medium urgency Display a persistent, in-app message that informs users of a particular change state.
Dialog Highest urgency Present alerts that require a user choice or acknowledgement and should interrupt the user's current task.
Page error Highest urgency Hides content on a page if users are not able to interact with content due to a server error.

Page errors

When an error occurs on the server and the user is not able to interact with content on a page, use the following patterns to communicate server errors. Provide a header with some information about the error, an explanation of what might be causing the error and the primary action the user might take to resolve an error. (This action might simply be “Refresh”).

Anatomy

Server error layouts can vary to support the type of content they contain. Server errors should always include an illustration and a brief description of the error. Other elements are optional.

1. Image (Required)
Select appropriate graphic from Forge Illustration library (coming). 

2. Header text (Optional)
Provide a brief description of what went wrong. Style should be conversational. For common error types, use the provided default copy (see below). 

3. Body text (Required)
 Describe the error in plain language and any steps user can take to fix the error. For common error types, use the provided default copy (see below). 

4. Up to two actions (Optional)
Provide up to two common actions users can take to recover or fix the error. For common error types, use the provided default actions (see below).

Faceted filtering

Faceted filtering enables users to search and pare down their results to match specific criteria.

Overview

Searching and filtering help users find specific information. Related, but distinct, searching enables a user to find a specific record or set of records, while filtering enables users to build a record set based on matched criteria. The two patterns can be used together to create a robust information-finding experience where users can search a general term and then iteratively refine their search. 

Searching versus filtering

Search

Use when:

  • The user has a specific record in mind.
  • The user is familiar with the record(s) they’re looking to find.

Use to:

  • Identify and locate a specific known record.
  • Display a specific set of results.

Filter

Filters are generally applied after an initial search.

Use when:

  • The user is looking to browse or explore a set of records.
  • The user wants to create a set of records based on specific criteria.
  • The user is looking to compare records with matching criteria.

Use to:

  • Allow the user to browse records for which they may not have a precise target in mind.
  • Narrow a set of search results to show a more targeted set.
  • Display a subset of records or information.
  • Create a context-specific record set.

Filters and facets

Filters enable users to find a set of records or items by excluding items that don’t meet certain criteria. They allow users to browse alternative sets when they don’t have a precise target in mind. Filters are context specific and should display categories that are relevant to the search at hand.

After an initial search has been completed or an initial dataset viewed, the set can refined using facets. Facets are more precise filtering dimensions within a filter category (ie, “Approved,” “Rejected,” “On Hold” in the category of “Status”). They allow users to refine results by several different dimensions simultaneously. Facets can provide “structure to help users understand the content space, and give them ideas about what is available and how to search for it” (Norman Nielsen Group, 2014).

Anatomy

A full page faceted search is comprised of five sections. 

  1. Search
  2. Filter side sheet
  3. Filters
  4. Search results header
  5. Search results body
  6. Search results footer

1. Search

Search allows a user to look for a specific record or set of records. 

The search field many incorporate any of the following patterns based on business requirements. 

  • Recent searches 
  • Popular searches 
  • Search suggestions 
  • Result suggestions
Recent and popular searches

Recent searches may be helpful in products where users search for things more than once or routinely search for the same or similar items. Recent search isn’t necessary for products where most searches tend to be novel. Recent searches appear as a dropdown when a user clicks into the search input, before they’ve started typing a query. Use the History icon to represent recent searches. 

Suggested searches such as popular searches, trending searches, etc may be used in place of or in conjunction with recent searches. Suggested searches should also appear when a user clicks into the search field.

 

Do.

Use the History icon for recent searches.

Do.

Use section labels when supporting multiple types of suggested search.

Search and result suggestions

Search may support two types of autocomplete suggestions.

Suggested searches offer search criteria that match a particular search, and can help users formulate effective queries. On click, the suggestion brings then to a set of search results. This pattern is best used for products where users are familiar with keywords but may not have a specific result in mind. Suggested searches are indicated by a Search icon.

Suggested results offer direct links to particular result detail pages. This pattern is best used when users are likely to have a specific search result in mind and know the criteria necessary to identify it.

Suggested searches and suggested results may be used independently or together.

Ranking

Autocomplete suggestions may be ranked by relevancy, popularity, recency, or other factors, depending on the business case.

Do.

Match words on leading letter combinations, unless there is a compelling business-case to match on middle or end sections of words. Use a medium font-weight on matching letters.

2. Filter side sheet

The filter side sheet contains facets used to pare down an initial search query. In general, users will perform a search and refine their search set with facets contained in the side sheet after their initial query. Conversely, products may display an initial set of records (ie, “most recent incidents”) that users may further refine with a set of facets.

Once a search has been performed or a set of results has been displayed, the filter side sheet should persist open by default. 

Filter side sheets are made up of three sections:

  • Title (required)
  • Subtitle section (required) 
  • Filters (required)

3. Filters

The filter side sheet contains filters and facets used to pare down an initial search query.

 

States

Filters have three states: default, expanded, and collapsed.

 

Types

Filters should follow specific formats based on what kind of content they contain.

 

Categories: Checkboxes

Checkboxes allow users to make selections from a list. They allow users to select more than one filtering criteria.

The dynamic counts to the right of the filter descriptions show the number of records matching the selected criteria.

Exclusive categories: Radio buttons

Use for non-overlapping categories where users can choose only a single option.

Date ranges: Date picker

Use to select an explicit date range.

Approximate range: Slider 
Sliders allow users to select from discrete value ranges. Sliders can be somewhat difficult to manipulate, so they should be used when an approximate value is good enough.

When using sliders, consider the unit. If only whole numbers can be chosen, for example, default the slider to use whole values only. Values on the left and right of the slider update to reflect the selected values.

Use “Reset” instead of clear.

Specific range: Input range 
Use for known, specific ranges. 

Use “Reset” instead of clear. 

4. Search results header

The search results header is comprised of:
  1. Applied filters (required)
  2. Total number of results (required)
  3. Ability to sort results (required)
  4. Toggle list type (optional)
  5. Ability to select all results (optional)
  6. Access to saved searches (optional) (guidance coming soon)
Default header

By default, a search results header contains:

  • Applied filters, displayed as removable chips 
  • Total number of results 
  • Sort order options
Displaying more filter chips

Filter chips take up the existing real estate to the left of the header’s additional options. Use a horizontal scroll to display additional active filters. 

Sort order options

Where users may want to view a list in two ways (ie, as a list and as a grid; as a list and on a map), toggle options may be displayed at the far end of the header. For more than three view options, use a menu.

Selection

When users can make a selection of a subset of records, place a “select all” control at the start of the header. For more information on selection, see “Selection” under Behavior. 

5. Search results body

The search results body displays a searched or filtered subset of records. Records can be displayed as a task cards, expandable task cards,  cards, or list items.

The search results body should have a max-width of 960px or 1056px on desktop.

A. Task cards

Use a task card to display a high level identifying information. Task cards may be used on their own or may be clicked to navigate to a detail page. Task cards are best used when records must be identified by a combination of more than two pieces of data.

By default, clicking a task card navigates to a detail page. Clickable task cards use an overlay on hover to indicate interactivity.

Task cards are comprised of 

  • An icon or avatar (optional, may be replaced with a checkbox as needed)
  • Primary identifying information (the data that users are most likely to use to identify a record)
  • Up to 5 pieces of secondary information (displayed as label-value pairs) 
  • Chips (optional, display up to two)
  • Actions (actions may be tucked away in a vertical overflow menu)

Default task cards.
Hover state on interactive task cards.

Displaying chips

Chips may be displayed to highlight differentiating or high-value information such as statuses. When chips are displayed, actions are tucked into the vertical overflow menu.

B. Expandable task cards

Use an expandable task card to display a few high level identifying pieces of information and allow users to expand for more information. Expandable task cards are best used when records can easily be identified by one or two specific pieces of data. 

Expandable task cards that expand should not open a detail page. Use a list or a task card instead. 

Expandable task cards are comprised of 

  • An icon, avatar, or checkbox (optional; icon or avatar may be replaced with a checkbox on hover as appropriate)
  • Primary identifying information (the data that users are most likely to use to identify a record)
  • Secondary information (optional) 
  • Chips (optional, display up to two)
  • Actions (up to two primary text actions with extras tucked into an overflow or up to three icon actions with extras tucked into an overflow menu)
  • Expansion region that may contain up to 10 label-value pairs OR custom information

Displaying chips

Primary actions may be replaced with chips, to highlight differentiating or high-value information such as statuses. When chips are displayed, actions are tucked into the vertical overflow menu.

C. Search results: Cards

Use cards to display data that features an image (like an employee profile) or to encourage browsing (like a Tyler app store).  Where scanability and comparison are important, opt for a list instead.

Search result cards are comprised of 

  • An image (optional)
  • Primary identifying information (the data that users are most likely to use to identify a record)
  • Secondary information (displayed as body text or label-value pairs) 
  • Actions (up to two; additional actions are tucked away in a vertical overflow menu)

Displaying matched criteria

Use a highlight to display matched criteria from a users’ search. Match on the specific query that a user entered.

6. Search results footer

Search results may use pagination or infinite scroll to display additional results. 

Pagination

Use pagination when records are sorted in a logical way (ie, chronologically or alphabetically). Pagination allows users to mentally bookmark or orient where they are in an ordered result set. Pagination should be used in the majority of backoffice Tyler apps. 

Infinite scroll

Infinite scroll is best used to encourage discovery or exploration when users are casually browsing content. Infinite scroll may also be a better option for mobile contexts. 

Behavior

Selection

Users may select a subset of records in order to perform actions on them. 

 

Entering selection mode 

On selection, the search results header transforms to a selection header. 

By default, the select all control selects currently visible records on the page. 

The selection header provides a summary of total records selected, the ability to select all available records, and any actions that can be taken on the selected set.

 

Exiting selection mode

Once actions have been taken, records revert to their default state. 

Applying filters

Updating filters: Desktop

By default, filters should be empty, unless the business case directs a logical default.

Use behavioral cues to dictate when to refresh search results. Allow users to update multiple criteria without interruption. Update search results after a mouse pause.

The main search results region should update, but the filter sidepanel should remain static so the user doesn’t lose their place within filters.

Display applied filters as dismissible chips above the search results area.

Updating filters: Mobile

On mobile, consider using a side sheet for filters (Nielsen Norman Group, 2015). A side sheet allows for continuous visibility of results so that users can still see some information about the results in the background of the filters.

Use an explicit “Filter” button to display filters.

  • Filters appear within a side sheet, with a scrim covering results in the background. Ensure that the total number of filters is always visible.
  • As with desktop, allow users to update multiple criteria without interruption and update search results after a mouse pause.
  • Clicking the X in the side sheet or tapping outside of the side sheet closes the side sheet and allows users to interact with search results.

What is Axe?

Axe is an open source accessibility library developed by Deque, an industry leader for accessibility testing. The open source library has inspired a lot of various projects to be built on top of it, including but not limited to Google Chrome’s lighthouse feature and Webhint.js (previously known as Sonarwhal).

How is Axe different from other accessibility libraries?

  • Works with various different testing suites like Jasmine, Qunit, Karma, Mocha, Maven, Rspec, Cucumber…etc.
  • Deque touts axe for not detecting false positives. “Axe only tests for accessibility issues that can be accurately detected via automation, and it only tests for components that actually exist on the page or application you’re testing.”
  • Widely supported and widely used “Axe is the most widely used accessibility testing library on the Internet. It has over a million downloads and has been adopted by Google and Microsoft for their own accessibility testing tools.”
  • Customizability

How is Socrata/Tyler D&I Division using it?

Automated Testing

Integration Tests (Cheetah)

For our Cheetah tests we use Rails’ Rspec library in order to conduct integration testing throughout our three different environment and various pages of our platform. There is ruby gem called axe-matchers that seamlessly integrates into our existing Rspec setup, allowing us to do automated accessibility testing throughout the frontend.

How to use axe-matchers

Adding accessibility testing to a page is as easy as adding the line:

expect(page).to be_accessible

This will cause axe to scan through the page and generate a list of issues

example below:

Failure/Error: expect(@page).to be_accessible Found 2 accessibility violations: 1) heading-order: Heading levels should only increase by one (moderate)https://dequeuniversity.com/rules/axe/3.1/heading-order?application=axeAPIThe following 2 nodes violate this rule: Selector: .calculation-panel-titleHTML: class="calculation-panel-title">Metric CalculationFix any of the following:- Heading order invalid Selector: .metric-preview > h5HTML:
Sample Result
Fix any of the following:- Heading order invalid

Testing specific elements on the page

axe-matchers does include an inclusion clause that allow you to specify a element you want to test

expect(page).to be_accessible.within '#selector1'

For more details check out this page

Excluding specific elements from testing

axe-matchers also allows you to exclude elements from testing

expect(page).to be_accessible.excluding '#selector1'

For more details check out this page

Thoughtful Integration

Our Cheetah test can be a handful to debug and work through at times. In order to not cause too much addition pain of having to debug and drop what you are doing to fix Cheetah tests due to accessibility, we are going with a more thoughtful integration approach. Slowly we will be adding specs page by page and fixing the various accessibility issues on that page. This way we can move the needle forward in terms of getting our frontend to be fully accessible while not randomizing existing team’s sprints and workflow to fix accessibility. As a part of the 2018 hackweek, tests were adding into the measure’s editing workflow and you can see them as an example (coming soon)

Unit Tests (Common Mocha)

The further upstream we can bring accessibility testing, the easier it is going to be. Aside from testing the entire page, there is also the option of testing the individual component with unit tests. axe-core, the main library, does have integration support for Mocha and Karma. There are definitely limitations regarding testing on a component level. For example, color contrast is something that can’t be tested until the component is put on the page since it depends largely on the component vs the background color. But any accessibility issues we can catch early on will greatly reduce the amount of cycles we will need to spend on accessibility.

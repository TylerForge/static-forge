What is accessibility? Accessibility allows a site‚Äôs content and functionality to be used by anyone. ‚ÄúAccessibility, then, refers to the experience of users who might be outside the narrow range of the ‚Äútypical‚Äù user, who might access or interact with things differently than you expect.‚Äù (Google design).¬†It concerns users who have some type of impairment or disability, whether permanent or temporary. Impairments include visual, audial, motor, hearing, and cognitive and can range from situational (ie, user is holding a baby while interacting with a tablet) to temporary (ie, user has had their eyes dilated) or permanent (user is visually impaired).

¬†

In general, utilizing accessibility best practices helps create applications that have a better experience for all users, not just those with impairments.

The¬†Web Content Accessibility Guidelines (WCAG) 2.0¬†are¬†a set of guidelines and best practices put together by accessibility experts to address what ‚Äúaccessibility‚Äù means in a methodical way ‚Äì they‚Äôre the standard we use to guide accessibility practices at Tyler.¬†

The WCAG is organized around four key principles:

  • Perceivable: Can users perceive the content? This helps us keep in mind that just because something is perceivable with one sense, such as sight, that doesn‚Äôt mean that all users can perceive it.
  • Operable: Can users use UI components and navigate the content? For example, something that requires a hover interaction cannot be operated by someone who can‚Äôt use a mouse or touch screen.
  • Understandable: Can users understand the content? Can users understand the interface and is it consistent enough to avoid confusion?
  • Robust: Can the content be consumed by a wide variety of user agents (browsers)? Does it work with assistive technology?

Aria labels

In general, Material and TCW components have ARIA labels and controls built in already. Check the Dev tools to see if your component has the appropriate aria roles and labels. Additional ARIA labels should be used for tools whose functions are not self apparent, or for controls that need extra context. Aria labels should be descriptive of the specific action to be taken; this is what the screen reader will read to the user.

Do.

Label icon buttons such as filter buttons.

Don‚Äôt.

Use vague or nonspecific aria-labels.

Focus

Focus determines where keyboard events go in the page at any time. Natively, Chrome applies a focus ring to focused elements. For users that operate a computer entirely with their keyboard, they may rely on focus to determine where they are on a page. HTML elements that naturally receive focus (radio-buttons, checkboxes, buttons, inputs) have built in focus-states; don‚Äôt remove these. Ensure that every interactive element can be tabbed to within your page and ensure that these elements have visually different focus states. Ensure that your page follows a natural and predictable tab order so that users can navigate your page via screen-reader or keyboard only.

Dialogs should naturally receive focus on their first input field when opened; mat-dialog does this inherently.

Do.

Use tabindex=‚Äù0‚Ä≥ to include elements in the natural tab order. Learn more about tabindex here.

Don‚Äôt.

Remove native border on focus.

Page structure

DOM order matters, not just for focus order but for screen reader order as well. When screen readers traverse a page, users have the option to navigate a page by its major sections or subheaders. Using native HTML semantic areas allows users to navigate to the content they care about more efficiently.

Screen readers use an accessibility tree to read through a page, so they‚Äôll follow the logical header order (H1 through H6) instead of the order seen visually on a page. Use headings to reflect page hierarchies, sections, and parent-child relationships, instead of for styling purposes. Label major sections with a semantic header instead of a DIV or SPAN.

Don‚Äôt.

Images

To provide non-sighted users a robust experience, provide alternate text to describe your images or graphs. Avoid using images of text to convey information; use HTML for this instead (the exception here would be branding or logos). Use alt tags to provide descriptions for images. Descriptions should not just be file names; use a short description that conveys in words what the image conveys visually. For images that do NOT provide information and should be skipped by a screen reader, use an empty alt tag: alt=‚Äù‚Äù.

Do.

Use descriptive text to convey information from an image or photo.

Don‚Äôt.

Use abbreviations or unclear alt tags.

Resizing

Your application should render gracefully both when zoom is increased to 200% and when the native browser font-size is increased. Content should not overlap nor should users need horizontal scrolling to read content. Use rem over pixels to ensure that your text plays nice with native browser font-size settings. rem calculates font-sizes responsively based on a root font size defined in the application. If the application has no root defined, the root size is likely 16px. Learn more about converting rem to px here. Angular Material components use px inherently; override these if you can. TCW components use rem.

Font size

Ensure that your font size is big enough for people to read. To quote a research report by Nielsen Norman,

Reduced visual acuity is probably the best-known aging problem, and yet websites with tiny type are legion. Sites that target seniors should use at least 12-point fonts as the default. And all sites, whether or not they specifically target seniors, should let users increase text size as desired‚Äîespecially if the site‚Äôs default is a small font size.

Design consistency
  • Ensure design consistency: Most senior citizens are generally aware of the standard colors for links and would usually assume an item with that color is a link. Avoid trying to implement new design colors for links and other active interaction elements. To quote the above Nielsen Norman report,¬†

Hypertext links are essential design components; using large text for them is especially important for two reasons: 1) to ensure readability, and 2) to make them more prominent targets for clicking. Also, you should avoid tightly clustered links; using white space to separate links decreases erroneous clicks and increases the speed at which users hit the correct link. This rule also applies to command buttons and other interaction objects, all of which should be reasonably large to facilitate easy clicking.

Contrast ratios

When using color for text, WCAG AA standards recommend a (minimum) contrast ratio of 4.5:1 for all text, with the exception for very large text. You can check your contrast here.

Guidance and best practices

Drag-able objects must be discoverable.

When moving objects is the primary task, show a static indicator that an object can be moved, like the¬†Reorder¬†icon, the¬†Drag handle¬†for 1-dimensional resizing. Provide instructional text if necessary.

Provide immediate feedback on hover or touch: a change in the object‚Äôs outline or coloring, or floating the object, all indicate the object is actionable. A change in cursor is also helpful; use the hand or move (four-arrow) cursor.

Dragging must be observable.

On mouse down or touch and hold:¬†Change in outline, highlight, or cursor. For desktop, can be an additional change from the on hover state.

On drag: Change in z-axis to illustrate lifting. Object sticks to cursor or touch point and tracks with movement.

The target area must be discoverable.

Static indications that a target area can receive an object: Intuitive layout and/or area labeling, explanatory hint text, instructional help text.

On hover interactions that indicate a target area can receive a dragged object: Hint text goes away, existing objects move to make room for the new addition, or target area becomes outlined and/or highlighted.

The result of dropping an object must be observable.

On mouse up: Change in z-axis and positioning of dragged object to illustrate placement. Object is no longer outlined or highlighted. Normal cursor. Target area reflects the addition of the new object.

The reorder icon can be displayed for as long as objects in either the source or target areas can be reordered.

If reordering is not a primary action, consider invoking a ‚ÄúReorder‚Äù or ‚ÄúMove‚Äù task mode where reordering is the primary or exclusive action.

Implementation

Here‚Äôs an example of how we used drag and drop on the Work Manager Calendar (EAM team) to allow users to schedule events by dragging them on and off the calendar interface.

Step One: Cursor hovers over the list item

Step two: List item ‚Äúwiggles‚Äù and shows a motion indicator via the icon on the left-hand side.

Step three: User grabs onto item, and a background develops underneath, as though you are pulling it away from a surface.

Step four: the item transforms into a smaller chip-like summary of the event, which is easier to track visually while dragging.¬†

Step five: the target area (in this case the box representing the 12th of November) develops a background, indicating to the user that they can drop the item onto the surface. The list where the item came from keeps the empty space where the dragged item originated until the item has been successfully dropped into another area.¬†

FAQ

What should the interaction of drag and drop look like? (When you select an event, should it remain as seen in the detail panel, or shrink to the calendar-sized box while you drag it)
  • Goals: large touch target size, only provide the information necessary to users. Immediate feedback once the action has been completed.¬†¬†
  • When doing a short click, the object must budge and then drop back to its place instantly.
  • When doing a long click, the object must move without actually moving the cursor and remain in its new position.
  • When the user selects the calendar detail panel, it should become slightly transparent and shrink to the size and appearance that it will appear as an event on the view of the calendar itself.¬†
  • The day of the calendar that you are about to drop the event on should have an obvious reaction to the user, so they know that they are dropping the event on that particular day. (This is exhibited by a color change in the mockup)
Is the drag and drop findable? Should we be using an indicator to suggest that functionality to users?
  • We should be using an indicator visible on the calendar, in addition to a different cursor when desktop users hover over it.¬†

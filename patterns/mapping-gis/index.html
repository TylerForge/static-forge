Mapping & GIS

Designing mapping interfaces are one of the more complex patterns. 

Guidance

There are several separate features incorporated in map interfaces. We will review some guidance for these here. 

  • Detail Panel
  • Hamburger Menu 
  • Layer/Legend Panel
  • Navigational Tools
  • Search Box
  • Toolbar

This is an oversimplification, but often there are two different aims when designing map interfaces, depending on the persona and scenario.

Back Office:

Built for power users and government employees who want to access lots of information and data via the map. Editable and searchable map focused on allowing user to access & manipulate data. Can exist as a standalone product or as an additional “panel” or feature in an existing program.

Citizen-Facing: 

Map intended to surface useful and relevant information to citizens and/or others.  Map focused on the presentation of data rather than its manipulation. May be a view-only map interface.

Note:

With maps, it’s tempting to include more functionality, to try and meet the needs of many people at once. Be cautious about this: the best map interfaces often start by trying to thoroughly meet the needs of one persona or user base.

Detail Panel

The detail panel is a sliding panel on the left-hand side that allows users to gain additional information about the map features they interact with.

When a user searches on something broad where there are multiple possible search results, the detail panel displays multiple search results, allows user to drill down into individuals or take action on any results. Quickfilters and tabs are optional.

When a user searches for a single element, or when a user drills down on search results, the detail panel shows additional information and allows users to take action. Also, the detail panel should never overlap the search.  

The detail panel is overlaid with filter criteria, for mapping applications that include filters. 

Note: For maps that exist as a panel (EAM’s Field Work or Asset Registry), and not in a standalone environment (Atlas), it may be appropriate to have selections on the map reflect changes in the program itself, rather than integrating a detail or list panel. This is a topic we are deciding on currently (see “search results and the most appropriate dynamic interactions on dual-panel programs”) 

Menu

The hamburger menu houses global settings and actions for your map interface. The hamburger icon should be on the upper left of the search bar.

Layer / Legend Panel

If you have a complex map with layers, you should combine layers & the legend into one panel.

 

Any individual features on the map should be links, when clicked they navigate the map to that feature. “Groups” and collections of features should not be links, but instead expand/collapse their child items. 

Checkboxes are the most intuitive method for showing that layers are on and off.

Sliders can be used on tablet and mobile devices, because of the naturally larger touch target. If you use checkboxes, just increase the touch target size and add the text title to the touch target, so clicking on the title of the layer will also turn it on or off. We also recommend including the title in the touch target for sliders. 

 

See the individual components for checkboxes and toggles for more information. 

 

Navigational Tools

Navigational tools are zoom in, zoom out, and current location. They should be reflected on every map interface. On responsive devices, these can be supplemented or replaced by gesture (pinch to zoom etc). They should be located on the bottom right of the map interface. 

 

Functions

  • Zoom in
  • Zoom out
  • Optional: Zoom “full”
  • Optional: “Current Location”

 

Search Bar

User research showed confusion amongst users regarding whether to use the omnibar to search the map. When we eliminated the omnibar and used an on-map search bar, this tested far better. Our recommendation is to turn off the “Search” in the omnibar, and include a search box on the upper left-hand corner of screen. *

The left-hand side of the search bar includes the hamburger menu if applicable. The search bar also has a search icon to the right.

 

*Some of the mockups shown here are older: so ignore the use of both the omni search and the search bar.

 

Future versions may also offer an “Advanced search” or additional querying mechanism here. See “upcoming decisions”
